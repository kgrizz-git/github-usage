## Bugs found (static review, not exercised by the test suite)

### High

1. **`src/github_usage/report_actions.py:88` — `TypeError: int + dict` in OS breakdown.**
   `os_millis` is the whole `{"UBUNTU": ..., "WINDOWS": ..., "MACOS": ...}` dict returned by `get_actions_from_runs`, not a per-OS scalar. `total_os[os_name] += os_millis` evaluates `int + dict` and raises `TypeError` on the first repo in the top 10 with positive minutes.
   *Fix:* `total_os[os_name] += os_millis[os_name]` (or change `get_actions_from_runs` to return per-OS minute values).

2. **`src/github_usage/billing.py:140` — `workflow_minutes` over-counts via cumulative `total_minutes`.**
   `total_minutes` is the cumulative sum across all runs, not the per-run minutes. After run N, the value added for the current workflow is the sum of every previous run's minutes as well, so per-workflow totals grow quadratically with multi-run repos. The "Top 3 Most Impactful Findings" output in `report_summary.py` will be wrong.
   *Fix:* compute a `run_minutes = sum(millis / 60000 for …)` for the current run and add it to `workflow_minutes[wf_name]`.

3. **`src/github_usage/auth.py:30-36` — `github.yaml` parser only matches column-0 `oauth_token:`.**
   The real `gh` CLI config is nested YAML where the key is indented (e.g. `  oauth_token: gho_…`). `line.startswith("oauth_token:")` only matches at column 0, so the fourth fallback never finds the token in a real `~/.config/github-cli/github.yaml`. The existing test (`tests/test_auth.py:35`) writes the key at column 0, masking the bug.
   *Fix:* match after stripping (`if line.lstrip().startswith("oauth_token:"):`) or use a real YAML parser; update the test fixture to use the indented format.

4. **`src/github_usage/api.py:43-46` — 403 rate-limit retry reads the wrong source.**
   `body.get("retry-after", 0)` reads the JSON body, but GitHub's secondary rate limit returns the delay in the `Retry-After` HTTP header, so the value is always 0. The retry branch is effectively dead code, and the recursive `self.request` has no max-retry counter.
   *Fix:* read `resp.getheader("Retry-After")` instead, and bound the recursion (e.g. max 3 retries).

### Medium

5. **`api.py:22-25` — Query string built without URL encoding.**
   `query = "&".join(f"{k}={v}" for k, v in params.items())` produces malformed URLs for values containing `&`, `=`, `+`, `#`, `?`, `/`, `:`, etc., and lets repo/branch names smuggle extra parameters.
   *Fix:* use `urllib.parse.urlencode(params)`.

6. **`api.py:60-77` — Pagination silently swallows non-list responses.**
   If the API returns an error envelope such as `{"message": "Bad credentials"}` (neither `[]` nor a list), the loop exits without raising. `get_all_pages` therefore masks auth/rate-limit errors and returns an empty list, which is then aggregated as "no data" further up. The "last page" detection (`len(result) < per_page`) also wastes one extra request when the final page is exactly `per_page` items long.
   *Fix:* inspect the `Link` header for `rel="next"`, and propagate error responses.

7. **`storage.py:10-11` — `KeyError` on malformed repo dict, no per-repo recovery.**
   `repo["owner"]["login"]` raises `KeyError` and aborts the whole storage analysis if any entry is missing `owner` or `name`. Other modules (`report_optional.py`) use `repo.get("owner", {}).get("login", "")` for the same purpose, and `get_storage_analysis` already has per-block `try/except RuntimeError` for the API calls but no such guard around the per-repo iteration.
   *Fix:* switch to `.get()` with defaults, and wrap the per-repo body in `try/except (KeyError, RuntimeError)`.

8. **`auth.py:40-50` — `check_user_scope` leaks connections and ignores status code.**
   The `HTTPSConnection` is never closed (no `try/finally`). A 401/403/404 from `/user` is treated identically to a 200, so a totally invalid token returns `False` for "user" scope and the caller prints the misleading "missing 'user' scope" message instead of "invalid token". Also, `X-OAuth-Scopes` is deprecated/absent for fine-grained PATs and GitHub Apps, so this will return `False` for tokens that *do* have user access.
   *Fix:* use `with` or `try/finally: conn.close()`; check `resp.status`; surface auth failures distinctly from missing-scope; consider `GET /user/installations` or the API rate-limit response for a scope-agnostic check.

9. **Inconsistent "No GitHub token found" follow-up wording across three call sites.**
   Verified across `cli.py:100-101` (email subcommand), `cli.py:179-182` (legacy default), and `legacy_report.py:33-36` (legacy entrypoint). The leading message is consistent, but the helper lines differ in wording (`Set GITHUB_TOKEN` vs `set GITHUB_TOKEN env var.`) and the command name (`github-usage` vs `./github-usage`). The test `test_missing_token_exits_one_with_clear_message` only checks for "No GitHub token found" and "GITHUB_TOKEN", so it does not pin down the exact wording.
   *Fix:* extract a single helper (e.g. `_print_missing_token(usage_command="github-usage")`) and use it from all three sites; tighten the test assertion.

10. **`report_data.py:142-162` — `get_warning_state` raises on `--warn-over N%` if Actions are skipped.**
    If a user runs `github-usage email-report --skip-actions --warn-over 80%`, the build raises `ValueError`, which is caught by `except (RuntimeError, ValueError)` in `cli.py:155` and the whole email run fails with a stack trace. The error is not gracefully surfaced as an "unavailable data" entry.
    *Fix:* return an empty warning list and add an explanatory message to `report["errors"]`, or document that `--warn-over N%` implies `--include-actions`.

11. **`api.py:21-58` and `legacy_report.py:50-58` — HTTP connections never closed.**
    Only `email_report.py:send_email` uses `try/finally: conn.close()`. `GitHubAPI.request` and the second `/user` call in `legacy_report.main` leave the underlying socket open until garbage-collected. In a long-lived legacy report run that issues many calls (per-repo billing, workflow runs, artifacts, releases) this is a slow TCP/FD leak.
    *Fix:* wrap each `request` body in `try/finally: conn.close()`, or hold a single `HTTPSConnection` per call and close it deterministically.

12. **`cli.py:79-82, 184-188` — `except SystemExit` swallows programmatic exits.**
    `int(exc.code or 0)` will raise `ValueError` if `exc.code` is a non-numeric string (e.g. `parser.exit("some message")`). It also catches *any* `SystemExit` raised inside `legacy_main`, including ones intended to terminate nested test harnesses. There is no test asserting that `parser.error` (e.g. bad `--max-repos foo`) returns a non-zero code cleanly.
    *Fix:* let `parser.parse_args` propagate `SystemExit` and only catch it for the `--help` case if translation is genuinely required; otherwise remove the `try/except` entirely.

### Low

13. **`report_summary.py` — file/function size violates the project's own style rule.**
    `AGENTS.md` says files > 300 lines or functions > 50 lines should be split. `report_summary.py` is 262 lines and `show_final_summary` is ~250 lines (lines 10-261). Hard to test or maintain; currently has zero unit tests.
    *Fix:* split into `print_cost_overview`, `print_top_consumers`, `print_storage_breakdown`, `print_utilization`, `print_findings`, `print_recommendations`.

14. **`report_summary.py:235` — `if X > 70 if Y else False:` mixes ternary with statement.**
    Mixing ternary expression with `if` statement is a maintenance hazard and the "70" threshold is a magic number.
    *Fix:* `if user_minutes and top2_sum / user_minutes * 100 > 70:`.

15. **`api.py:32-37` — JSON parse failure on success returns `{}` silently.**
    A 200 with a non-JSON body (e.g. GitHub returning an HTML error page during an outage) is indistinguishable from a successful empty response. Downstream code that does `data.get("usageItems", [])` then sees "no items" rather than "API problem".
    *Fix:* log the body length and the first 200 chars, or raise `RuntimeError` so callers can surface the error.

16. **`report_optional.py:44, 65` — `int(...)` raises on malformed sizes.**
    `sum(int(item.get("size_in_bytes", 0)) for item in artifacts)` and the analogous `releases` line raise `ValueError` if the API ever returns a non-integer (e.g. a string `"1024"` or a `float`); this propagates out of the per-repo loop. Neither function has a `try/except` around the int conversion, and tests don't exercise this.
    *Fix:* `int(item.get("size_in_bytes") or 0)` or skip the item if conversion fails.

17. **`billing.py:121-141` — `os_minutes` name lies (it holds millis).**
    `os_minutes = {"UBUNTU": 0, …}` is incremented with `os_minutes[os_name] += millis`. The dict holds raw millis, not minutes; the caller has to remember to divide by 60 000. The variable name will mislead future readers and is the root cause of bug #1.
    *Fix:* rename to `os_millis` and have the caller convert, or change the function to return minutes.

18. **`cli.py:175-189` and `_resolve_email_token` — `sys.argv` mutation is process-global.**
    Both `main()` and `_resolve_email_token()` save and restore `sys.argv`, but the mutation is not thread-safe. Anyone running the library under `concurrent.futures` (or with a signal handler that inspects `sys.argv`) will see torn state. The `finally` block does help with normal use.
    *Fix:* pass argv explicitly to `resolve_token(token=...)` or use an explicit argument rather than `sys.argv`.

19. **`api.py:14-19` — `User-Agent` is fixed.**
    `User-Agent: github-usage-report-v3` is a single static string. GitHub's API will accept it, but some endpoints (and rate-limit heuristics) prefer a contact-style UA. Minor; only a suggestion.

20. **`email_report.py:14, 19-25` — `_generated_line` can print `Generated: None`.**
    `day = generated_at[:10] if generated_at else datetime.now(tz=UTC).date().isoformat()` — if `data.get("generated_at")` is `None` (only reachable via custom callers; `build_report_data` always sets it), `str(None)[:10]` is `"None"`, which then fails `fromisoformat` and prints `Generated: None` (with a stray leading space). The `str(data.get("generated_at", ""))` at line 44 in `format_report_email` makes this a concrete code path.
    *Fix:* if `generated_at` is falsy, use today's date directly.
