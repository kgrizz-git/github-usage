## Phase 1 — Correctness & Display

- [x] **Fix #1: Broken token pre-check in `_run_legacy_report`** (`cli.py:282`).
  `resolve_token(argv=legacy_argv)` passes a list that starts with `"github-usage"` (the program name), so the token resolution always returns a truthy string, bypassing the fail-fast gate. Replace the call with a direct check on the peeled `token` variable, or remove the pre-check entirely since `legacy_main()` independently resolves the token and handles the missing-token case.
  - **Test:** Add a test in `tests/test_cli.py` that verifies `_run_legacy_report` exits with code 1 when no token is available (mocking `resolve_token` to return `None` from env/gh auth).

- [x] **Fix #5: Progress bar overflow when usage exceeds free-tier limits** (`report_summary.py:163-170`).
  When `min_pct > 100`, `min_filled > bar_len`, and the bar extends beyond the expected 40-character width. Clamp `min_pct` to `min(100, min_pct)` before computing `min_filled`, and do the same for `storage_pct`.
  - **Test:** Add a test in `tests/test_report_summary.py` that verifies the progress bar caps at 40 characters when usage exceeds 100%.

- [x] **Fix #6: CSV column order depends on dict key insertion order** (`export_csv.py:159-163`).
  `_write_nested` uses `sample.keys()` for the header row but `sku_data.values()` for data rows. If any SKU dict has keys inserted in a different order, columns misalign. Extract a canonical column list from the first item and use it for consistent value lookup on all subsequent items.
  - **Test:** Add a happy-path test that verifies correct output when all dicts share the same key order (typical of the GitHub API), plus a test with intentionally different key orders that verifies column alignment is correct regardless.

- [x] **Fix #7: Redundant fallback `/user` call is dead code** (`cli.py:307-309`).
  When `legacy_main()` returns `None`, `_run_legacy_report` makes a fresh `GET /user` call as a fallback. `legacy_main()` (in `legacy_report.py`) has exactly one return site (`return username` at line 128) and calls `sys.exit(1)` on all error paths (RuntimeError, KeyboardInterrupt, missing token). In `cli.py` the `SystemExit` is caught at line 301–302 and returns early, so the `username = legacy_main(...)` assignment only runs when `legacy_main` succeeds — at which point `username` is always a non-empty string. The fallback is genuinely unreachable. Remove it and its associated test.

  After removing the fallback, add a defensive `or "unknown"` guard on the `username` assignment (e.g., `username = legacy_main(...) or "unknown"`) so that if `show_account_info` ever returns an empty string, `build_report_data` still receives a valid value rather than propagating `None`.

  - **Test:** Remove `test_export_calls_user_when_legacy_main_returns_none` from `tests/test_cli.py`. Confirm `test_export_does_not_call_user_when_legacy_main_returns_username` still passes.

## Phase 2 — Performance / API Economy

- [x] **Fix #3: `get_all_pages` fetches all repos before truncating** (`api.py:99-137`, `report_data.py:141-143`).
  `_limited_repos` and `legacy_report.py:79` both call `get_all_pages("/user/repos", ...)` which paginates through every repository the user has, then the caller slices (or uses all of them, in the legacy case). Add an optional `limit` parameter to `get_all_pages` that short-circuits pagination when the limit is reached. Apply it at both call sites.
  - **Test:** Add a test in `tests/test_api.py` that verifies `get_all_pages` respects the new `limit` parameter and stops paginating early. Verify the two repo-listing callers in `report_data.py` and `legacy_report.py` pass an appropriate limit when one is available.

## Phase 3 — Data Accuracy

- [x] **Fix #4: Zero-values from failed API calls indistinguishable from actual zero** (`billing.py:96-103`, `report_optional.py:29-32`).
  Two files need changing:

  **`billing.py` — introduce `BillingFetchError` and stop swallowing it.**
  `get_actions_per_repo` currently catches `RuntimeError` and returns `(0.0, 0.0, {})`, hiding failures completely. Add a `BillingFetchError(RuntimeError)` exception class (in `billing.py` or a shared `exceptions.py`) and replace the silent `return 0.0, 0.0, {}` with `raise BillingFetchError(f"{owner}/{repo}: {e}")`.

  **`report_optional.py` — repurpose the dead `except` block and expose errors.**
  The `try/except RuntimeError: continue` wrapper in `get_repo_consumers` (lines 29–32) is currently dead code — it never fires because `get_actions_per_repo` swallows the exception first. After the `billing.py` change it becomes live. Change it to `except BillingFetchError as e: errors[full_name] = str(e)` and add an `errors: dict` to `get_repo_consumers`'s return value alongside `by_minutes` and `by_cost`. Existing tests in `tests/test_report_optional.py` that mock `get_actions_per_repo` returning `(0.0, 0.0, {})` on failure will need to be updated to raise `BillingFetchError` instead.

  - **Test — error path:** Add a test that verifies `get_repo_consumers` records an entry in `errors` for repos whose billing endpoint raises `BillingFetchError`, rather than silently including them with zero usage.
  - **Test — success path:** Add a test that verifies repos with genuinely zero usage (API returns empty/zero items successfully, `get_actions_per_repo` returns `(0.0, 0.0, {})` without raising) are recorded as zero-usage entries in `by_minutes` / `by_cost`, not flagged as errors. This ensures the fix doesn't over-report.
