## 1. Broken token pre-check in `_run_legacy_report`

**File:** `src/github_usage/cli.py:282`
**Severity:** Low (defensive gate is bypassed; real error surfaces downstream anyway)

The line `resolve_token(argv=legacy_argv)` reconstructs an argv list that starts with `"github-usage"` (the program name). Since `resolve_token` (in `auth.py:27`) treats `argv[0]` as the token, it returns `"github-usage"` — a truthy string — making the pre-check succeed every time even when no real token is available.

```python
# cli.py:281-282
legacy_argv = ["github-usage", *([token] if token else []), *flag_argv]
if not resolve_token(argv=legacy_argv):
```

The actual token resolution happens later inside `legacy_main()`, which calls `resolve_token()` with no args (using `sys.argv[1:]`). So the bug doesn't cause crashes — it just renders the fail-fast gate useless.

**Fix:** Either pass only the peeled token to `resolve_token`, or skip the pre-check entirely and let `legacy_main` handle it.

## 2. Double API calls when exporting from legacy report

**File:** `src/github_usage/cli.py:259-334`
**Severity:** Medium (doubles API quota consumption; no correctness impact)

When `--export` is used with the legacy interactive report:

1. `legacy_main()` fetches all billing data for the terminal display.
2. Then `_run_legacy_report` calls `report_data.build_report_data()` to fetch the *same data again* for export.

Every billing endpoint, repo listing, and per-repo Actions call is duplicated. For users with many repos, this can exhaust rate limits quickly.

**Fix:** Either (a) have `legacy_main` return the collected data so the export can reuse it, or (b) have the export path call `build_report_data` once and feed the results to both the display and export writers.

## 3. `get_all_pages` fetches all repos before truncating to `max_repos`

**Files:** `src/github_usage/api.py:99-137`, `src/github_usage/report_data.py:141-143`
**Severity:** Medium (performance/rate-limit waste for users with many repos)

`_limited_repos` calls `api.get_all_pages("/user/repos", {"type": "all"})`, which paginates through *every* repository the user has access to, then slices to `max_repos`:

```python
# report_data.py:141-143
def _limited_repos(api, max_repos):
    repos = api.get_all_pages("/user/repos", {"type": "all"})
    return repos[:max_repos], len(repos) > max_repos
```

A user with 5,000 repos requesting `--max-repos 10` still makes ~50 paginated API calls to fetch all 5,000 repos. Since `get_all_pages` has no early-termination mechanism, this is unavoidable with the current design.

**Fix:** Add a `limit` parameter to `get_all_pages` that short-circuits pagination when the limit is reached, or use the `per_page` parameter combined with a manual page loop.

## 4. Failed per-repo Actions API calls silently produce zero-usage entries

**Files:** `src/github_usage/billing.py:96-103`, `src/github_usage/report_optional.py:29-32`
**Severity:** Low (data accuracy)

When `get_actions_per_repo` catches a `RuntimeError`, it returns `(0.0, 0.0, {})`:

```python
# billing.py:96-103
def get_actions_per_repo(api, owner, repo):
    try:
        billing = api.request(...)
    except RuntimeError:
        return 0.0, 0.0, {}
```

The caller `get_repo_consumers` treats this identically to a repo with genuinely zero Actions usage. A repo whose billing endpoint failed (e.g., due to a permissions issue) will incorrectly appear as using 0 minutes and $0.00, with no indication that data is missing.

**Fix:** Return a sentinel or raise a distinct exception so the caller can distinguish between "no usage" and "failed to fetch."

## 5. Progress bar overflow when usage exceeds free-tier limits

**File:** `src/github_usage/report_summary.py:163-170`
**Severity:** Low (cosmetic display issue)

```python
# report_summary.py:165-168
min_pct = (user_minutes / free_min_limit * 100) if user_minutes else 0
bar_len = 40
min_filled = int(min_pct / 100 * bar_len)
print(f"    {'█' * min_filled}{'░' * (bar_len - min_filled)}")
```

When usage exceeds 100% (e.g., 3000 of 2000 free minutes), `min_pct = 150`, `min_filled = 60`, and `bar_len - min_filled = -20`. Python treats `'░' * -20` as `""`, so the bar silently extends beyond the expected 40-char width instead of capping at 100%.

**Fix:** Clamp `min_pct` to `min(100, min_pct)` before computing `min_filled`.

## 6. Column misalignment risk in CSV SKU breakdown writer

**File:** `src/github_usage/export_csv.py:159-163`
**Severity:** Low (relies on consistent API response key ordering)

```python
def _write_nested(writer, section_name, row_label, nested):
    sample = next(iter(nested.values()), None)
    if isinstance(sample, dict):
        writer.writerow([row_label, *sample.keys()])
        for key, sku_data in nested.items():
            writer.writerow([key, *sku_data.values()])
```

The header row uses `sample.keys()` but subsequent rows use `sku_data.values()`. If any SKU dict has keys in a different insertion order than `sample`, the columns will be misaligned. In practice, all items from the same API response should share the same key order (Python 3.7+ dicts are insertion-ordered), but this is a fragile assumption.

**Fix:** Extract a canonical column list from the first item's keys and use it to look up values in each subsequent item rather than relying on `.values()` order.

## 7. `default_subject` with empty-string `generated_at`

**File:** `src/github_usage/email_report.py:11-14`
**Severity:** Low (never triggered in practice, but fragile)

```python
def default_subject(username, generated_at=None):
    day = generated_at[:10] if generated_at else datetime.now(tz=UTC).date().isoformat()
```

If `generated_at` were ever an empty string `""`, the `if generated_at` check would be `True` (empty strings are falsy — wait, `""` is falsy, so the else branch runs). Actually, this is correct: both `None` and `""` trigger the fallback. No bug here — retracting.

**Verdict:** Not a bug. The falsy check correctly handles both `None` and `""`.

## 8. `_run_legacy_report` fallback `/user` call may be redundant

**File:** `src/github_usage/cli.py:307-309`
**Severity:** Low (extra API call, no correctness issue)

```python
if not username:  # fallback when legacy_main exited without returning one
    user = api.request("GET", "/user")
    username = user.get("login") or "unknown"
```

`legacy_main()` already calls `show_account_info(api)` which does `GET /user` and returns the username. When `legacy_main` returns `None` (which happens on error/exit), the code makes a fresh `GET /user` call. In the normal path where `legacy_main` returns the username, this block is skipped. But the path where `legacy_main` returns `None` typically means a fatal error occurred — making another API call at that point is unlikely to succeed.

**Fix:** Consider storing the resolved username earlier in the flow rather than relying on `legacy_main`'s return value.
