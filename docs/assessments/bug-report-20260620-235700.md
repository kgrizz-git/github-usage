## Bugs found (static review, not exercised by the test suite)

### High

#### A1. `src/github_usage/report_actions.py:43-44, 83-84` â€” `KeyError` on malformed repo dict (regression of bug #7)

`show_actions_per_repo` and `show_actions_os_breakdown` both read
`repo["owner"]["login"]` and `repo["name"]` directly. The same pattern was
fixed in `storage.py` (bug #7, see `storage.py:11-12`) and in
`report_optional.py:26-27, 55-56, 80-81` (which use `.get(..., "")`), but
`report_actions.py` was missed.

Reproduction (verified locally):

```python
from unittest import mock
from github_usage.report_actions import show_actions_per_repo, show_actions_os_breakdown

api = mock.Mock()
repos = [
    {"owner": {"login": "octocat"}, "name": "valid"},
    {"name": "missing-owner"},   # crashes in both functions
    {},
    {"owner": {}},               # crashes (no login, no name)
]

with mock.patch("github_usage.report_actions.get_actions_per_repo",
                return_value=(0.0, 0.0, {})):
    show_actions_per_repo(api, repos)
    # KeyError: 'owner' (after printing the first valid row)

with mock.patch("github_usage.report_actions.get_actions_from_runs",
                return_value=(0.0, {"UBUNTU": 0, "WINDOWS": 0, "MACOS": 0}, {})):
    show_actions_os_breakdown(api, repos)
    # KeyError: 'owner' (no output before crash)
```

Both functions are reached from `legacy_report.main` via
`repos = api.get_all_pages("/user/repos", {"type": "all"})` (line 79) â†’
`show_actions_per_repo(api, repos)` (line 80) and `show_actions_os_breakdown(api, repos)` (line 82).
A single malformed repo in the user's list (e.g., a deleted repo whose
`owner` field is null) aborts the entire legacy report with a traceback
before the rest of the report is printed. The `storage.py` and
`report_optional.py` paths use defensive `.get()` and survive the same
input; the report-actions path does not.

*Fix:* mirror `storage.py:10-15` â€” wrap the per-repo body in
`try/except (KeyError, RuntimeError)` and read fields via
`repo.get("owner", {}).get("login")` and `repo.get("name")`. Add a
`test_show_actions_per_repo_skips_malformed_repos` test analogous to
`tests/test_storage.py::test_get_storage_analysis_handles_malformed_repos`.

#### A2. `src/github_usage/export_xlsx.py:32-40` â€” header styling applied to the wrong row

In `write_sheet`:

```python
ws.append([_safe_cell("=" * 50)])    # row 1
ws.append([f" {title}"])              # row 2
ws.append([_safe_cell("=" * 50)])    # row 3
for row in rows:
    ws.append([_safe_cell(cell) for cell in row])   # row 4+
for cell in ws[3]:                    # <- styles the second '=' separator
    cell.font = header_font
    cell.fill = header_fill
```

`ws[3]` is the **third** row, which contains the second `===` separator,
not the actual column-header row. Verified by inspecting the saved
workbook (cell A3 has the blue `4472C4` fill; cell A4 does not):

```
Row 1: [("'==================================================", None)]
Row 2: [(' Report Metadata', None)]
Row 3: [("'==================================================", None)]   <- blue fill here
Row 4: [('Username', 'octocat')]                                          <- no fill
```

The `header_font` (bold white) and `header_fill` (blue) are intended for
the column-header data row (row 4 in the layout above, the first row of
the `rows` list). No test inspects fill colors, so the regression
survived CI.

*Fix:* change `for cell in ws[3]` to `for cell in ws[4]` (and rename the
`header_font`/`header_fill` to make the intent explicit). Optionally
collapse the layout to a single separator and apply the styling to the
actual data rows.

#### A3. `src/github_usage/cli.py:252-326` (`_run_legacy_report`) â€” third `/user` call wastes rate-limit budget per run

`legacy_main.main` already calls `/user` twice (once in
`report_account.show_account_info` at line 51, once in
`auth.check_user_scope` at line 59) and returns the resolved username.
`_run_legacy_report` then makes a **third** call to `/user` at
`cli.py:300` purely to recover the username for the export step:

```python
if export_format and export_format != "none":
    token = resolve_token(argv=legacy_argv)
    api = GitHubAPI(token)
    user = api.request("GET", "/user")      # third /user call
    username = user.get("login") or "unknown"
    data = report_data.build_report_data(api, username, ...)
```

For a typical `--export json` run on a free-tier token with 60
req/hour, this consumes 5% of the budget just for the username lookup.
`legacy_main` already returns the username; capturing it would eliminate
the extra call.

*Fix:* capture the return value of `legacy_main(...)` and pass it to
`build_report_data` (fall back to fetching `/user` only if the return is
`None`).

  `_write_consumers`, `_write_artifact_storage`, `_write_release_assets`,
  `_write_insights`, `_write_errors`) would bring each writer under 100

  and `src/github_usage/cli.py:_run_email_report` (101 lines). Both are
  linear sequential sections; the same per-section extraction would
  apply.

None of these is a runtime bug, but the project's own
`scripts/check-sizes` already flags them, and the
`docs-check` stale-assessment reminder (added 2026-06-19 per
`TO_DO.md`) makes the threshold enforcement one of the things the
maintenance checklist tracks.

#### A5. `src/github_usage/report_optional.py:9-18` â€” `_safe_int_size` silently truncates floats and accepts booleans

```python
def _safe_int_size(value) -> int | None:
    if value is None:
        return None
    try:
        return int(value)
    except (ValueError, TypeError):
        return None
```

`int(True)` returns `1`; `int(1024.7)` returns `1024`. The float
truncation is documented by the existing test
(`test_truncates_float_with_fraction`), so it is intentional, but
booleans are not addressed. A release asset whose `size` field is
`true` would be counted as `1` byte, which is almost certainly not the
intended value but not a hard failure either. Low risk because the
GitHub API returns integer bytes, but a guard (`if isinstance(value, bool): return None`)
would make the helper unambiguous and align the docstring ("for
numeric inputs") with the runtime behavior.

#### A6. `src/github_usage/report_data.py:146-167` â€” `_single_warning_state` raises `ValueError` on garbage `--warn-over` values

```python
def _single_warning_state(report_data: dict, warn_over: str) -> list[str]:
    raw = warn_over.strip().removeprefix("$")
    if raw.endswith("%"):
        ...
        threshold = float(raw[:-1])          # ValueError if "abc%"
    threshold = float(raw)                    # ValueError if "abc"

```

`ValueError` is caught one level up by `cli._run_email_report`'s

`Error: invalid --warn-over value 'abc' (expected a dollar amount like 50 or $50, or a percentage like 80%)`.
Functional, but the error path is rough. Low priority.
