# Report Export Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Add export support so users can save GitHub usage reports as CSV, XLSX, PDF, JSON, or plain text files. The CLI adds `--export FORMAT`, `--output PATH`, `--json`, `--no-interactive`, and `--dry-run` flags. Exported files go through a redaction layer before writing. A planned `--month YYYY-MM` flag for historical billing queries is **deferred**: see `docs/api-discovery-month.md` for the discovery results and rollback rationale.

**Architecture:** Add an `export_report.py` module with format-specific writers and a shared redaction layer. Integrate export into both the legacy report flow and the email-report flow. Use stdlib for CSV and JSON; add optional dependencies for XLSX and PDF behind feature flags so the base package stays dependency-free.

---

## Design Decisions

| Decision | Choice | Rationale |
|----------|--------|-----------|
| XLSX support | Optional dependency (`openpyxl`) via `export-xlsx` extra | Keeps base package dependency-free; XLSX is popular for billing data |
| PDF support | Optional dependency (`fpdf2>=2.7`) via `export-pdf` extra | fpdf2 2.7+ provides `XPos`/`YPos` enums required by the PDF writer |
| CSV support | stdlib `csv` module | Always available, no extra dependency |
| JSON support | stdlib `json` module | Already needed for `--json` flag |
| Text support | stdlib file I/O | Reuses existing plain-text formatters |
| Redaction scope | File exports only | Interactive terminal and email body show full data; email export files use `redact_data=True` |
| Month flag | `--month YYYY-MM` on both report commands | **Deferred.** API discovery (see `docs/api-discovery-month.md`) confirmed GitHub's billing endpoints ignore `since`/`until` parameters, so historical queries are not feasible. The flag is not added to the CLI in this PR. |
| Writer interface | All writers accept `(data, file_obj)`; orchestrator owns path, atomic write, stdout | Avoids the "open file twice" bug from passing both path and file object |
| Atomic write | Orchestrator writes to `tempfile.mkstemp`, passes file object to writer, then `os.replace`s to the final path | Prevents partial/corrupted files on failure |
| CSV BOM | Always written (UTF-8 BOM at start) | Excel-on-Windows requires BOM for non-ASCII; trade-off accepted (slight `cat` annoyance for the major use case) |
| CSV section header | `### Section Name ###` (3 hash chars per side) | Short, single-cell, single-line; matches the example output |
| None handling | CSV/JSON/XLSX: empty cell/string; PDF: `"N/A"` | Spreadsheets/JSON handle empty cells gracefully; PDF needs a visible label for the reader |
| `--email-format` vs `--export` | Email-report uses `--email-format text\|html` for body; `--export` for file | Disambiguates two text-related flags; `--email-format` does not affect file output |
| Last day of month | `calendar.monthrange(year, month)[1]` | Standard library handles February in leap and non-leap years |

## CLI Interface

### New flags for the legacy report (`github-usage [GITHUB_TOKEN]`)

```
github-usage [GITHUB_TOKEN] \
  [--export csv|xlsx|pdf|json|text|none] \
  [--output PATH] \
  [--json] \
  [--no-interactive] \
  [--dry-run]
```

| Flag | Type | Default | Description |
|------|------|---------|-------------|
| `--export FORMAT` | `choice` | `none` | Export format. Values: `csv`, `xlsx`, `pdf`, `json`, `text`, `none`. In interactive mode with a TTY and no `--export` set, prompt the user. |
| `--output PATH` | `str` | `<auto-generated>` | Output file path. When omitted with `--export` set, generates a filename like `github-usage-YYYY-MM-DD.csv`. |
| `--json` | `store_true` | `False` | Shorthand for `--export json`. **Mutually exclusive with `--export`**; if both are set, error. Prints JSON to stdout when no `--output` is set. |
| `--no-interactive` | `store_true` | `False` | Never prompt; use defaults. Required for CI/script usage. |
| `--dry-run` | `store_true` | `False` | **No-op for the legacy flow** (legacy always prints to stdout). Use `--export json --output -` to inspect structured data, or `--json` for stdout JSON. |

> `--month YYYY-MM` is **deferred** — see `docs/api-discovery-month.md`.

### New flags for `email-report` subcommand

```
github-usage email-report \
  [--export csv|xlsx|pdf|json|text|none] \
  [--output PATH] \
  [--email-format text|html]
```

| Flag | Type | Default | Description |
|------|------|---------|-------------|
| `--export FORMAT` | `choice` | `none` | Same as legacy report. Email reports default to `text` export when `--output` is set. |
| `--output PATH` | `str` | N/A | Reuses the same export path infrastructure. |
| `--email-format text\|html` | `choice` | `text` | Render format for the email body. `--email-format` only controls the email body; file output format is controlled by `--export`. HTML support is deferred. |

### Behavior matrix

| Scenario | Behavior |
|----------|----------|
| Interactive, no `--export`, no `--no-interactive` | Prompt user for export format (or skip) |
| Interactive, `--export none` | No file output, same as current behavior |
| Non-interactive or `--export <format>` | Write export file silently |
| `--json` without `--output` | Print JSON to stdout |
| `--output` without `--export` and without `--json` | Error: `--output requires --export or --json` |
| `--month` historical queries | **Deferred** — GitHub's billing endpoints ignore `since`/`until` (see `docs/api-discovery-month.md`). The flag is not in the CLI. |
| Missing optional dependency (e.g., `--export xlsx` without `openpyxl`) | Error: `pip install github-usage[export-xlsx]` |
| `--json` with `--export` (any value, including `none`) | Error: `--json and --export are mutually exclusive` |
| `--email-format text` with `--export csv` | No conflict: `--email-format` controls email body, `--export` controls file output |
| `--dry-run` (legacy) | No-op |
| `--dry-run` (email-report) | Print email body to stdout; skip Resend send |

### Redaction behavior

| Output target | Redaction applied? | Details |
|---------------|-------------------|---------|
| Interactive terminal | No | Full data shown to user |
| File export (`--export csv/json/xlsx/pdf/text`) | Yes | Structured fields redacted by key; text patterns (emails, dollar amounts) redacted |
| Email body (sent via Resend) | No | Full data sent to recipient; redaction only applies to file attachments |
| Email export file (`--export text --output`) | Yes | Same as file export redaction; email-report passes `redact_data=True` when `--output` is set |

## Architecture

```
cli.py
  |
  +-- legacy report flow
  |     +-- resolve_token()
  |     +-- GitHubAPI(token)
  |     +-- check_user_scope(api)
  |     +-- legacy_report.main(month=..., export=..., output=..., no_interactive=..., dry_run=...)
  |           +-- terminal.show_*()       -> stdout (interactive display, unchanged)
  |           +-- report_data.build_report_data(api, username, month=...) -> dict  <-- NEW
  |           +-- export_report.export(data, export_format, output_path, redact_data=True) -> file
  |
  +-- email-report flow
        +-- _email_parser() parses flags
        +-- _resolve_email_token()
        +-- GitHubAPI(token)
        +-- check_user_scope(api)
        +-- report_data.build_report_data(api, username, month=..., ...)
        +-- email_report.format_report_email(data, month=..., body_format="text") -> str
        +-- if --export <format> and --output set:
        |     export_report.export(body, export_format, output_path, redact_data=True) -> file
        +-- if --export <format> and no --output: skip file export
        +-- if --dry-run: print(body) to stdout (skip Resend send)
        +-- else: email_report.send_email(...)
```

### `export_report.py` — Export orchestrator

**Responsibility:** Owns file paths, atomic writes, stdout routing, dependency checks, and redaction. Writers are pure (data in, file object out).

```python
def export(
    data: str | dict,
    export_format: str,           # csv | xlsx | pdf | json | text | none
    output_path: str | None = None,
    username: str | None = None,
    month: str | None = None,
    redact_data: bool = True,
    to_stdout: bool = False,
) -> str:
    """Return the absolute path of the written file, or '' for stdout/no-op."""
```

Behavior:
- If `export_format == "none"`, return `""`.
- Validate that `csv`/`xlsx`/`pdf` receive dict data, not strings (raise `ValueError` otherwise).
- Check optional dependencies (`openpyxl`, `fpdf2`) before writing; raise `RuntimeError` with install hint if missing.
- Apply redaction to dict (`redact.redact_report_data`) or string (`redact.redact_text`) if `redact_data=True`.
- For file output: create a `tempfile.mkstemp` in the target directory, pass the file object to `writer.write(data, file_obj)`, then `os.replace` to the final path. On exception, unlink the temp file.
- For stdout output: pass `sys.stdout` to `writer.write`.
- Auto-generate the filename via `_generate_filename(export_format, username, month)` when `output_path` is None.

### Writer contract (applies to all five writers)

```python
def write(data: str | dict, file_obj) -> None:
    """Write `data` to `file_obj` (a real file or sys.stdout)."""
```

- Writers do **not** open files or check dependencies — the orchestrator handles both.
- Writers do **not** apply redaction — the orchestrator handles it.
- All section getters must coalesce `None` to `{}` or `[]` (e.g., `data.get("repo_consumers") or {}`), since `build_report_data` sets unpopulated sections to `None` (`src/github_usage/report_data.py:234-241`).
- The `api_estimate` key is part of `ReportData`; JSON preserves it as-is, other formats ignore it.

### `redact.py` — Redaction layer

```python
REDACT_USERNAME = "[redacted-user]"
REDACT_REPO = "[redacted-repo]"
REDACT_EMAIL = "[redacted-email]"
REDACT_AMOUNT = "[redacted-amount]"

def redact_report_data(data: dict) -> dict: ...   # deep copy, redacts by key
def redact_text(text: str) -> str: ...             # regex: email + dollar amounts only
```

Invariants:
- **Username redaction** is by key (`data["username"]`). No generic regex pattern, to avoid false positives on SKU names / model names.
- **Repo redaction** is by key (`{"repo": ...}` entries anywhere in the dict, recursively). Auto-covers future sections.
- **Text redaction** matches email and dollar-amount regexes only. Username redaction is skipped for text (too many false positives on ordinary words).
- `redact_report_data` returns a deep copy; the input is never mutated.
- Idempotent: re-running on already-redacted text produces the same result.

### `export_csv.py` — CSV writer

- Section header format: `### Section Name ###` (3 hash chars on each side, single cell, single line).
- Always writes UTF-8 BOM (`\ufeff`) at the start of the file.
- `None` values are written as empty cells.
- Dict values are stringified automatically by `csv.writer` (no manual conversion).
- Sub-tables (SKU breakdown, Copilot by model) get a header row then data rows.
- Trailing empty row for POSIX compliance.

### `export_json.py` — JSON writer

- `json.dumps(..., indent=2, ensure_ascii=False, cls=_DatetimeEncoder)`.
- `_DatetimeEncoder` handles `datetime.datetime` and `datetime.date` (`.isoformat()`).
- Trailing newline for POSIX compliance.
- Preserves all `ReportData` keys, including `api_estimate`.

### `export_text.py` — Text writer

- `write(data, file_obj)` — if `data` is a string, write directly; if dict, call `email_report.format_report_email(data)` then write.
- **Deferred import** of `email_report` to prevent a circular dependency if `email_report` ever imports from `export_report`.
- Trailing newline appended if missing.

### `export_xlsx.py` — XLSX writer

- One sheet per major section. Sheet names truncated to 31 chars (Excel limit).
- `_safe_cell(value)` helper prevents Excel formula injection: prefixes values starting with `=`, `+`, `-`, `@` with a single quote.
- `None` values are written as empty cells.
- Stdout output uses `sys.stdout.buffer` (binary mode; required for XLSX).

### `export_pdf.py` — PDF writer

- Requires `fpdf2>=2.7` (uses `XPos`/`YPos` enums, e.g., `new_x=XPos.LMARGIN, new_y=YPos.NEXT`).
- Cover page (title + metadata), then one page per section.
- `_MAX_SECTION_ROWS = 30`; excess rows are truncated with a `(truncated: N more rows)` note.
- `_fmt_num(value)` casts numeric values via `float()` and falls back to `str()` for string-formatted numbers.
- `None` values display as `"N/A"`.
- Stdout output uses `sys.stdout.buffer` (binary mode; required for PDF).

## Data Mapping for Export Formats

### CSV structure

One CSV file with multiple sections. Each section is preceded by a header row of the form `### SECTION_NAME ###` (single cell, single line, 3 hash chars per side). Within each section, data is written as key-value rows or sub-tables.

Example (abbreviated):

```
### Report Metadata ###
username,octocat
period,current_month
generated_at,2026-06-15T14:30:00Z

### Actions Usage ###
minutes,1250.0
minutes_limit,2000
minutes_percent,62.5

### Copilot By Model ###
model,requests,gross,discount,net
gpt-4.1,30.0,0.0,0.0,0.0
claude-sonnet-4,12.0,0.0,0.0,0.0

### Unavailable Data ###
error_key,error_message
copilot,Token may lack billing scope.
```

Full section list (in order):
1. Report Metadata (username, period, generated_at)
2. Warnings (one per row)
3. Actions Usage (minutes, limits, storage)
4. Actions SKU Breakdown (sub-table: SKU, minutes, storage_gb_hours, gross, discount, net)
5. Copilot Usage (totals)
6. Copilot By Model (sub-table: model, requests, gross, discount, net)
7. Git LFS (totals)
8. Monthly Costs (per category: gross, discount, net)
9. Top Repos by Minutes (repo, minutes, gross, storage_avg_mb)
10. Top Repos by Cost (repo, minutes, gross, storage_avg_mb)
11. Artifact Storage (repo, artifact_bytes)
12. Release Assets (repo, release_asset_bytes)
13. Key Insights (one per row)
14. Unavailable Data (error_key, error_message)

### JSON structure

The full `ReportData` dict as produced by `build_report_data()`, serialized with `json.dumps(..., indent=2, ensure_ascii=False, cls=_DatetimeEncoder)`. The `api_estimate` key is preserved. Datetime values are emitted as ISO-8601 strings.

### XLSX structure

One sheet per section in the CSV section list above. Each sheet has a decorative header (decorative `===` rows wrapped in `_safe_cell`), followed by key-value pairs or tabular data.

### PDF structure

Multi-page PDF with a cover page (title, username, period, generated_at) and one page per section from the CSV list. Sections exceeding `_MAX_SECTION_ROWS = 30` are truncated with a `(truncated: N more rows)` note.

## Interactive Prompt Behavior

When the user runs `github-usage` in an interactive terminal (TTY detected) without `--export` or `--no-interactive`, prompt before the report runs:

```
Export options:
  1) CSV   - Spreadsheet-compatible comma-separated values
  2) XLSX  - Excel workbook (requires openpyxl)
  3) PDF   - Formatted PDF document (requires fpdf2)
  4) JSON  - Machine-readable JSON
  5) Text  - Plain text report
  6) None  - No export file

Choose an export format [None]: _
```

- Default is `None` (no file output).
- If the user selects a format with missing optional deps, show the install instruction and return to the prompt.
- If the user selects a file format, ask for output path (default: auto-generated filename).
- If the path exists, prompt: `File already exists: /path/to/file.csv. Overwrite? [y/N]: _`
- If `--month` is specified, the default filename includes the billing month: `github-usage-octocat-2026-05.csv`.
- Pressing enter accepts the default.
- `--no-interactive` skips all prompts and uses defaults (no export, no path prompt).
- `--output` flag bypasses the path prompt and writes directly to the specified path (no overwrite check).
- `--json` writes JSON to stdout by default; if `--output <path>` is also specified, JSON is written to the file instead of stdout.
- `--export none` is the same as omitting `--export` (no export file produced).
- `--json` and `--export` are mutually exclusive; using both raises an error.

## `--month YYYY-MM` — Deferred

API discovery has been completed; see `docs/api-discovery-month.md` for the
full results. Summary: all four tested billing endpoints
(`/users/{username}/settings/billing/usage/summary?product=*` and
`/users/{username}/settings/billing/premium_request/usage?product=copilot`)
**ignore** the `since`/`until` parameters and return the same shape with or
without them. Historical month queries are therefore not feasible with the
current GitHub API.

**Rollback executed:** the `--month` flag is not added to the CLI in this PR.
The flag can be reintroduced if GitHub adds date-range support to these
endpoints in the future. Run the discovery script again with

```
GITHUB_USAGE_API_DISCOVERY=1 GITHUB_TOKEN=<token> \
    python3 -m github_usage.scripts.api_discovery_month
```

to refresh the findings.

The `legacy_report.main()` signature still accepts a `month` parameter (for
forward compatibility and to keep the type stable), but the CLI never sets
it.

## Deferred Enhancements

These are intentionally out of scope for the first export implementation:

- `--month YYYY-MM` historical billing queries (see `docs/api-discovery-month.md`).
- `--email-format html` for email report body rendering.
- CC/BCC delivery fields for email reports.
- Report retention to GitHub Releases, S3, or shared drives.
- Caching or persisting artifact/release snapshots for time-series comparison.
- Redaction of usernames in text export (only applied to structured data to avoid over-redaction).
- Configurable redaction rules (current plan uses fixed patterns).
- `--timeout` and `--max-retries` for API calls during historical month queries.
- PDF headers/footers with page numbers.
- XLSX cell styling (bold headers, column widths, conditional formatting).
